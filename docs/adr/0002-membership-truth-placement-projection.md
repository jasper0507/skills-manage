# 成员关系为真源；盒内不落第二套位置

分类工作台用盒子收纳占位时，若同时持久化「盒/隔间成员列表」与「占位 Location=在某盒」，两份数据会分叉（在列表却显示在桌面、或声称在盒却不在任何列表）。E2 已用 Open 修补（rehome），但写路径仍双写，修补不能从模型上消灭分叉。

**决定：**

1. **成员关系**（普通盒或组合盒隔间内的占位 id 集合）是「谁在盒里」的唯一真相；**一枚占位至多一处**成员关系。多盒归类靠 **复制占位**（CONTEXT 已定），不靠一枚 id 多挂。
2. **占位安置**只持久化两种非盒状态：**桌面**（网格行列）与 **回收站站内**。盒内占位 **不**再持久化一套可与成员关系冲突的「在盒」位置。
3. **工作台对外视图**（桌面/HTTP）仍可把成员关系 **投影** 成「在盒」位置，供 UI 与现有契约阅读；投影是视图，不是第二真相。
4. 读入旧索引时 **以成员关系为准** 规范化；无成员却声称在盒 → 落到桌面空位；非回收且无合法桌面坐标 → 视口内空位。不 bump schema 版本，靠 Open/保存规范化演进。

**未选：** 继续双写 + 仅靠 Open rehome（分叉会复发）；以 Location 反推成员列表（与成员真源相反）；一枚占位多处挂载（与桌面隐喻和复制多归类冲突）。

产品语言见 `CONTEXT.md`（成员关系、占位安置）。
